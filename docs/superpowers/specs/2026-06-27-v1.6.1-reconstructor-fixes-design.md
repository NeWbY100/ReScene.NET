# v1.6.1 — RAR Reconstructor & Settings Fixes (Design)

**Date:** 2026-06-27
**Status:** Approved (pending implementation plan)
**Scope:** Four independent bug fixes, app-only (no `ReScene.Lib` changes). Released as v1.6.1.

## Background

User feedback after v1.6.0 surfaced four issues. A fifth (clean-temp) and a sixth (AcoustID)
were parked. The Stop-hang was root-caused via systematic debugging: the data loss is confirmed
(the Output cleanup deletes the Release folder when they're the same), and the hang is a
downstream effect of that same invalid configuration; normal cancellation traces as working.
The chosen fix is a validation guard (prevents the data loss and removes the hang trigger), not a
speculative cancellation rewrite.

## Fixes

### Fix 1 — Reject Release == Output (data loss + hang trigger)

**Problem:** In `ReconstructorViewModel.StartAsync`, the "output folder not empty → delete
contents" cleanup runs without checking that Output differs from Release. When they are the same
folder (or one is nested in the other), the cleanup deletes the release's input files; the run
then errors, and the pathological state can wedge Stop so the process must be killed.

**Design:**
- Add a pure helper to `ReconstructorFieldGuidance`:
  ```csharp
  /// <summary>
  /// True when two folder paths are the same folder, or one is nested inside the other —
  /// configurations where deleting the output folder's contents would also remove the release.
  /// </summary>
  public static bool PathsOverlap(string pathA, string pathB)
  {
      if (string.IsNullOrWhiteSpace(pathA) || string.IsNullOrWhiteSpace(pathB))
      {
          return false;
      }

      string a, b;
      try
      {
          a = AppendSeparator(Path.GetFullPath(pathA));
          b = AppendSeparator(Path.GetFullPath(pathB));
      }
      catch (Exception ex) when (ex is ArgumentException or PathTooLongException or NotSupportedException)
      {
          return false; // Unparseable paths fall through to the existing per-path validation.
      }

      return a.Equals(b, StringComparison.OrdinalIgnoreCase)
          || a.StartsWith(b, StringComparison.OrdinalIgnoreCase)
          || b.StartsWith(a, StringComparison.OrdinalIgnoreCase);
  }

  private static string AppendSeparator(string path) =>
      path.EndsWith(Path.DirectorySeparatorChar) ? path : path + Path.DirectorySeparatorChar;
  ```
- In `StartAsync`, after the Release-exists validation and before the output create/cleanup block,
  add:
  ```csharp
  if (ReconstructorFieldGuidance.PathsOverlap(ReleasePath, OutputPath))
  {
      Log(LogTarget.System, "Output folder overlaps the release folder.");
      _fileDialog.ShowError("Validation Error",
          "The Output folder must be different from the Release folder, and not inside it.");
      return;
  }
  ```
  Placing it before any deletion guarantees nothing is wiped.

**Tests:** `PathsOverlap` — equal paths → true; output nested in release → true; release nested in
output → true; siblings/unrelated → false; case-insensitive match → true; trailing-separator
variants → true; empty input → false.

### Fix 2 — Creators embed the running version, not a stale persisted one

**Problem:** `AppSettings.DefaultAppName` is persisted. It seeds from the live assembly version
only on first run; once saved, it never refreshes, so after an upgrade the SRR/SRS creators embed
e.g. `"ReScene.NET v1.5.0 (92f6e9f)"`. (The title bar/About read the live assembly version
directly, which is why they are correct.)

**Design:**
- Add a pure helper to `FormatUtilities`:
  ```csharp
  /// <summary>
  /// Returns the effective default app name: the live <see cref="GetDefaultAppName"/> when the
  /// stored value is blank or an auto-generated "ReScene.NET v…" string (so it refreshes across
  /// upgrades), otherwise the user's custom value unchanged.
  /// </summary>
  public static string NormalizeAppName(string? stored)
  {
      if (string.IsNullOrWhiteSpace(stored)
          || stored.StartsWith("ReScene.NET v", StringComparison.Ordinal))
      {
          return GetDefaultAppName();
      }

      return stored;
  }
  ```
- In `AppSettingsService.Load()`, after the settings object is obtained and before returning, set
  `settings.DefaultAppName = FormatUtilities.NormalizeAppName(settings.DefaultAppName);`. This
  auto-heals a stale persisted value on every load while preserving a real custom app name.

**Tests:** `NormalizeAppName` — a `"ReScene.NET v1.5.0 (abc)"` input returns `GetDefaultAppName()`
and is not equal to the stale input; a blank/null input returns `GetDefaultAppName()`; a custom
value (e.g. `"MyGroup"`) is returned unchanged.

### Fix 3 — "Verify" field copy says `.sfv`/`.sha1`, not `.srr`

**Problem:** The verification field only accepts `.sfv`/`.sha1` (Start rejects anything else; the
dialog filter is already correct), but several strings call it an `.srr` file.

**Design (copy only):**
- `ReScene.NET/Views/ReconstructorView.xaml` — Verify description run:
  `— The .srr file to verify reconstructed archives against.` →
  `— The .sfv or .sha1 file to verify reconstructed archives against.`
- `ReScene.NET/ViewModels/Reconstruction/ReconstructorFieldGuidance.cs`:
  - doc comment `Status for the verification (.srr/.sfv/.sha1) file path.` →
    `Status for the verification (.sfv/.sha1) file path.`
  - Required hint `Required — choose the .srr/.sfv/.sha1 to verify against.` →
    `Required — choose the .sfv or .sha1 to verify against.`
  - error `This .srr file does not exist.` → `This .sfv/.sha1 file does not exist.`

No filter change (`FileDialogFilters.VerificationFiles` already lists SFV/SHA1 only).

### Fix 4 — Settings WinRAR-versions folder propagates without a restart

**Problem:** `ReconstructorViewModel` pulls the WinRAR (and Output) path from settings only in its
constructor (via `ApplyPathDefaultsFromSettings`, only-if-empty). Changing the setting later does
not propagate until the app restarts.

**Design:**
- In the `ReconstructorViewModel` constructor, when `_settingsService` is non-null, subscribe to
  `_settingsService.Changed` and, on change, call `ApplyPathDefaultsFromSettings()` followed by
  `RefreshPathStatuses()`. This mirrors the `CreatorViewModel`/`SRSCreatorViewModel` pattern.
- Keep the existing only-if-empty rule inside `ApplyPathDefaultsFromSettings`, so a newly-set
  WinRAR/Output folder fills a blank field immediately but never overwrites a path the user typed.
- The `Changed` event is raised from `AppSettingsService.Save()` on the UI thread (Settings is
  saved from the UI), so the handler runs on the UI thread; setting the path properties drives the
  existing status hooks normally.

**Tests:** if the test doubles provide a fake `IAppSettingsService` that can raise `Changed` and
return updated settings, add a VM test: with an empty `WinRarPath`, raising `Changed` after the
fake's settings gain a WinRAR path fills `WinRarPath`. Otherwise rely on the clean build plus the
manual check (the mechanism mirrors the already-working Creators).

## Non-Goals

- No cancellation-path rewrite (guard removes the known hang trigger; deeper hardening deferred
  unless a hang reproduces afterward).
- No clean-temp feature, no AcoustID.
- No `ReScene.Lib` change (app-only v1.6.1).
