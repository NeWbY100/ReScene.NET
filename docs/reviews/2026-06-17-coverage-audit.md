# ReScene.NET — Code Review & Unit-Test Coverage Audit

_Generated 2026-06-17 via a 20-unit fan-out review with adversarial verification of every finding (40 agents). Read-only static audit; tests were not executed._

**Totals:** 200 confirmed findings (7 critical, 54 high, 91 medium, 48 low) — 4 findings rejected by the verifier as false positives, out of 204 raw, across 20 review units.

## Part A — Genuine defects (not just missing tests)

These are confirmed product bugs / robustness defects the audit surfaced. Ordered by severity.

### 🔴 [critical] BitInput silently truncates any compressed input larger than 32KB (MaxSize); file-body decompression over 32KB is wrong and untested  `(rar-decompression)`
- **Where:** `ReScene.Lib/ReScene/RAR/Decompression/BitInput.cs` :: BitInput.SetBuffer (lines 128-138), MaxSize const (line 12); consumed by RARArchive.TryReadAllBytes -> RARDecompressor.Decompress (RARArchive.cs:254-268)
- **What:** InBuf is a fixed 0x8000 (32KB) buffer and SetBuffer copies only Math.Min(length, MaxSize) bytes, zero-padding the rest. GetBits returns 0 once InAddr+2 >= InBuf.Length. For comments this is fine (tiny), but RARArchive.TryReadAllBytes feeds the ENTIRE packed file body (up to maxUnpackedBytes, default 8MB) through this same path. Any compressed entry whose packed data exceeds 32KB will be silently truncated: the decoder reads zeros past 32KB and the result is wrong (typically short/zero-padded). This is a latent correctness bug in reconstruction of real (non-trivial) archive members.
- **Evidence:** BitInput.cs line 12: const int MaxSize = 0x8000. SetBuffer line 135: Array.Copy(..., Math.Min(length, MaxSize)). RARArchive.cs line 254-268 reads packed = new byte[entry.PackedSize] and passes the whole thing to RARDecompressor.Decompress. Every test input (comments, test_wrar40_m5 file body) is well under 32KB, so the truncation branch is never exercised. No test uses a >32KB compressed payload.
- **Regression test:** Add a test that decompresses an archive whose single compressed entry has PackedSize > 0x8000 and assert the full byte-for-byte content matches a known-good stored copy (mirror TryReadAllBytes_Rar5_M3_DecompressesToStoredRar4Equivalent but with a >32KB body). It should currently FAIL/return truncated data, documenting the bug. Also add a direct BitInput test: SetBuffer with a 40000-byte array, advance InAddr past 0x8000 via AddBits, and assert behavior is defined (currently GetBits returns 0 past the cap).

### 🟠 [high] RAR5 parse loop has no exception guard; DateTime.FromFileTime on corrupt FILETIME throws out of Parse()  `(rar-detailed)`
- **Where:** `ReScene.Lib/ReScene/RAR/RARDetailedHeader.cs` :: ParseRAR5 (lines 1011-1031); DateTime.FromFileTime at lines 1620 and 2073 (AddTimeField, Metadata creation time)
- **What:** ParseRAR4 wraps each block parse in try/catch (lines 342-384) so malformed input degrades gracefully, but ParseRAR5 (the while loop calling ParseRAR5Block) has NO try/catch. DateTime.FromFileTime(ft) throws ArgumentOutOfRangeException when the 64-bit value is negative or represents a date outside DateTime range. This is reachable from real/corrupt RAR5 data via the Metadata extra record creation-time branch (line 1619-1620) and via AddTimeField for non-Unix HTIME File Time records (line 2072-2073). A binary-inspection tool fed corrupt scene data will throw straight out of the public Parse(Stream)/Parse(string) methods instead of returning the blocks parsed so far.
- **Evidence:** Source: line 1619 `long ft = reader.ReadInt64(); var dt = DateTime.FromFileTime(ft);` and line 2072-2073 identical pattern in AddTimeField — both inside RAR5 extra-area parsing which runs under ParseRAR5Block, and ParseRAR5 has no catch. Contrast RAR4 ParseRAR4 lines 376-384 which catches and advances. Tests: no test feeds malformed RAR5 data at all; the only malformed-input tests are Parse_EmptyStream and Parse_StreamTooShortForSignature which never reach block parsing.
- **Regression test:** Build a synthetic RAR5 file with a Main Archive header containing a Metadata extra record (fieldType=2) whose flags set CTIME(0x02) but clear UNIXTIME(0x04), and a creation-time payload of 8 bytes = 0xFFFFFFFFFFFFFFFF (or 0x8000000000000000). Assert that RARDetailedParser.Parse(stream) does NOT throw and returns the signature + main header blocks. Also add the same for an HTIME (File Time, fieldType=3) record with non-Unix flags and an out-of-range FILETIME. The fix is to wrap FromFileTime in try/catch (or guard the value) and/or add a per-block try/catch in ParseRAR5 mirroring ParseRAR4.

### 🟠 [high] FileModifiedTimes matching uses a filename read from the wrong offset on LARGE-flag headers  `(rar-patcher)`
- **Where:** `ReScene.Lib/ReScene/RAR/RARPatcher.cs` :: PatchStream filename extraction (lines 477-482) feeding TryPatchFileMtime (line 519)
- **What:** fileName is extracted as ASCII starting at fixed offset 32. For a file header with the LARGE flag, the FILE_NAME actually begins at offset 40 (32 base + 8 HIGH bytes). So on LARGE headers fileName contains the HIGH_PACK_SIZE/HIGH_UNP_SIZE bytes followed by a truncated name. The dictionary lookup in TryPatchFileMtime (targetMtimes.TryGetValue(fileName,...)) will therefore never match the real file name for any file >2GB (exactly the case where LARGE is set), so per-file mtime patching silently no-ops on large files. The test suite even acknowledges the corrupted name at lines 846-847 but only works around it instead of treating it as a defect.
- **Evidence:** Line 481: fileName = Encoding.ASCII.GetString(fullHeader, 32, Math.Min(nameSize, headerSize - 32)); — no adjustment for (flags & Large). TryPatchFileMtime correctly computes extTimeOffset = 32 + (hasLarge?8:0) + nameSize (line 345), proving the author knows the name is offset by 8 when LARGE is set, yet the name string passed for matching is read at 32 regardless. No test exercises FileModifiedTimes against a LARGE-flag header.
- **Regression test:** Build a RAR4 file header WITH the LARGE flag and an EXT_TIME mtime region, file name 'subs.idx'. Set FileModifiedTimes={"subs.idx"->target}. Call PatchFile. Assert the DOS FTIME and EXT_TIME remainder were rewritten and the block re-parses with ModifiedTime==target. With the current code this test will fail (name mismatch), demonstrating the bug; the fix is to read the name from 32+(hasLarge?8:0).

### 🟠 [high] ParseRarFileBlock reads name length with no bounds check — unhandled EndOfStreamException on a truncated RARFile block  `(srr-parser)`
- **Where:** `ReScene.Lib/ReScene/SRR/SRRFileParser.cs` :: ParseRarFileBlock (lines 171-193)
- **What:** Unlike every other Parse*Block method, ParseRarFileBlock does NOT validate `fs.Position + 2 <= headerEnd` before calling `reader.ReadUInt16()` for the filename length, and it has no headerEnd/headerSize bounds derivation at all. Load only guarantees `headerSize >= 7` and `blockEndPos (= startPos + headerSize + addSize) <= fs.Length`. A RARFile block (0x71) with headerSize exactly 7 and no LongBlock flag (addSize=0) located at the very end of the file passes all of Load's guards, then ParseRarFileBlock reads 2 bytes past EOF and throws EndOfStreamException, which propagates out of SRRFile.Load uncaught. Real-world truncated/incomplete SRR downloads are explicitly a supported scenario (see incomplete_srr fixtures), so this crashes on plausibly malformed input.
- **Evidence:** ParseRarFileBlock starts with `ushort nameLen = reader.ReadUInt16();` (line 174) with no prior `fs.Position + 2 > headerEnd` check, whereas ParseStoredFileBlock (line 137), ParseOSOHashBlock (line 58), ParseRarPaddingBlock (line 91) and ParseHeaderBlock (line 30) all guard the 2-byte name-length read. Load validates headerSize>=7 (line 455) and blockEndPos<=fs.Length (line 461) but never that >=2 bytes of name-length data exist within the block. No test constructs a truncated 0x71 block; the only truncation coverage is the well-formed incomplete_srr fixtures which still have complete RARFile headers.
- **Regression test:** Add Load_TruncatedRarFileBlock_DoesNotThrow: write SRR header (7 bytes) + a RARFile block consisting of exactly the 7 base bytes (crc=0x7171, type=0x71, flags=0x0000, headerSize=7) and nothing after it, write to file, call SRRFile.Load. Assert it does not throw (currently throws EndOfStreamException) and that srr.HeaderBlock is non-null / srr.RARFiles is empty. Then add the matching bounds check to ParseRarFileBlock so the block is skipped/returns null.

### 🟠 [high] WMVContainerRebuilder never restores stripped Data Object payload, producing corrupt output  `(srs-rebuilder)`
- **Where:** `ReScene.Lib/ReScene/SRS/Rebuilders/WMVContainerRebuilder.cs` :: Rebuild (whole method), esp. the 'Copy body verbatim' branch (lines 63-68)
- **What:** The WMV writer (WMVContainerHandler.WriteSRS) keeps the ASF Data Object's original 24-byte header (with its original full objSize) but explicitly STRIPS the packet payload ('Data packets are stripped', WMVContainerHandler.cs line 161), recording the payload only as track signature/DataLength. The rebuilder, however, treats every non-SRS-GUID object identically: it writes the 24-byte header then does CopyBytes(srsFs, outFs, totalSize-24) reading the body verbatim FROM THE SRS. For the Data Object those payload bytes do not exist in the SRS (they were stripped); the bytes physically following the Data Object header in the SRS are the 26-byte data header followed by the injected SRSF/SRST GUID blocks. The rebuilder will therefore copy the wrong bytes as the data-object body, never read mediaFilePath/trackOffsets at all, desync the object walk, and produce output that can never byte-match the original. mediaFilePath and trackOffsets parameters are entirely unused, which is itself a red flag for a format whose payload is stored only in the media file.
- **Evidence:** WMVContainerRebuilder.Rebuild signature receives mediaFilePath and trackOffsets but the method body references neither; it only opens srsFs and outFs and copies bodies verbatim from srsFs. Meanwhile WMVContainerHandler.WriteSRS line 149 writes the original header (objSize unchanged) and lines 161-162 strip the packet payload. No restore-from-media logic exists in the rebuilder.
- **Regression test:** Add a WMV synthetic builder (minimal ASF: Header Object + a Data Object with a known file-ID GUID 3626B275..., totalPackets, and N fixed-size packets of >=256 random bytes). Then: (a) SRSWriter.CreateAsync over it, (b) SRSRebuilder.RebuildAsync(srs, sample, out), (c) assert result.Success && result.CRCMatch && byte-for-byte File equality with the original. This test should currently FAIL and pin the bug. Also add a separate-larger-media variant where the Data Object packets are embedded in a bigger ASF file.

### 🟡 [medium] ReadServiceBlockData mutates stream position without restoring it, with no test pinning the contract  `(rar4-headers)`

### 🟡 [medium] PeekBlockType can throw EndOfStreamException on truncated multi-byte vints despite the 6-byte guard  `(rar5-headers)`
- **Where:** `ReScene.Lib/ReScene/RAR/RAR5HeaderReader.cs` :: PeekBlockType() lines 484-506
- **What:** PeekBlockType only checks `Position + 6 > Length` then unconditionally calls ReadVInt twice (header-size and type) after skipping 4 CRC bytes. - **Evidence:** PeekBlockType_InsufficientData covers only the <6-byte (returns null) case. No test feeds 6+ bytes whose header-size vint has a continuation bit with no following byte. ReadVInt throws EndOfStreamException (proven by ReadVInt_TruncatedMultiByte) and PeekBlockType does not catch it.

At minimum document the throw with an Assert.Throws test so the behavior is pinned.

### 🟡 [medium] SerializeSrsf silently truncates app/file name length field on names > 65535 UTF-8 bytes  `(srs-file-writer)`

### ⚪ [low] ReadVInt silently truncates the 10th byte instead of detecting overflow  `(rar5-headers)`
- **Where:** `ReScene.Lib/ReScene/RAR/RAR5HeaderReader.cs` :: ReadVInt() lines 514-537
- **What:** The overflow guard is `if (shift > 63) throw`. After reading the 9th byte shift becomes 63, which is NOT > 63, so a 10th byte is read and OR'd in at `<< 63`. Only bit 0 of that byte's 7 data bits lands in bit 63 of the ulong; bits 1-6 are shifted to positions 64-69 and silently discarded with no error. The InvalidDataException only fires when an 11th byte is present. So a malformed/oversized vint with high bits set yields a wrong value rather than throwing, corrupting header-size, data-size, name-length or unpacked-size parsing for adversarial/corrupt scene data.
- **Evidence:** Trace: byte0 shift0->7, ... byte8 shift56->63 (63 not >63, loop continues), byte9 read with `result |= (b & 0x7F) << 63` then shift63->70 only thrown if a byte10 follows. ## Part B — Coverage gaps & findings by subsystem

### `rar4-headers` [lib] — 9 findings (0C / 2H / 4M / 3L)

**Coverage assessment:** RAR4 header parsing is well covered overall. RARHeaderReaderTests + RARFlagsTests + RARUtilsTests give strong coverage of base-header reading, truncation guards, header-size guards, header-CRC validation (valid + corrupted for archive/file/service/end blocks), LARGE 64-bit size combining, service-block (CMT/RR/AV) sub-type/data parsing, DOS-date conversion edge cases, dictionary-size extraction, and RAR Unicode filename decoding (modes 0/1/2). Indirect end-to-end coverage via SRRFileTests exercises file/directory headers, split flags, and extended-time ctime/atime reading (the i!=0 separate-DOS-time branch); RARArchiveTests run ReadBlock/SkipBlock/ReadServiceBlockData over real WinRAR-generated RAR files. The notable gaps are all in the HIGH-PRECISION extended-time path and a few error/boundary branches: extended times with >0 extra precision bytes (PrecisionFromExtraBytes 1/2/3, TryReadRemainder count>0, the rmode &0x4 +1-second correction) are never parsed from bytes; the Salt-skip path is never exercised; truncated/malformed extended-time and salt mid-header inputs are untested; and SkipBlock's includeData-adds-AddSize branch and over-length clamp branch lack direct unit tests. RARServiceBlockInfo's ExtTime precision branch in ParseServiceBlock is also untested.

**Test-quality notes:** Test quality is generally good: real WinRAR RAR files are used (RARArchiveTests), CRC is computed with the same Force.Crc32 the SUT uses (slight tautology risk, but acceptable since it verifies offset/range, not the algorithm), and LARGE-size tests assert exact 64-bit composed values rather than just "not null". A few weak spots: ReadBlock_FileHeader_ParsesDOSTimestamp only asserts ModifiedTime is NotNull (no exact value), so an off-by-one in mtime would slip through at the reader level (RARUtilsTests do assert exact values for DosDateToDateTime separately, so this is partially mitigated). The extended-time end-to-end tests in SRRFileTests assert only ContainsKey (presence), never the actual decoded ctime/atime DateTime values or precision, so wrong fractional/second math would not be caught. No test asserts stream Position after ReadServiceBlockData or after ReadBlock, so the documented "indeterminate position" contract is unverified.

#### 🟠 High-precision extended-time parsing (extra bytes >0) is completely untested
- _high · untested-edge-case · confirmed_ — `ReScene.Lib/ReScene/RAR/RARHeaderReader.cs` :: ReadExtendedTimes / TryReadRemainder / PrecisionFromExtraBytes (lines 554-651, 845-852)
- Every test that sets the ExtTime flag uses rmode nibble 0x8 (present, 0 extra bytes). The branches that read 1/2/3 fractional precision bytes and map them to HighPrecision1/HighPrecision2/NtfsPrecision are never executed from actual byte input, only the enum values are asserted in RARFlagsTests. This is the most complex bit-twiddling in the unit and directly affects bit-exact RAR reconstruction (mtime/ctime/atime sub-second precision).
- **Test:** Build a file header with ExtTime where mtime rmode nibble = 0xB (present + 3 extra bytes, NTFS precision) and 3 fractional bytes, e.g. extFlags high nibble 0xB and remainder bytes [0x00,0x00,0x80] -> 0x800000 ticks. Assert FileHeader.MtimePrecision == NtfsPrecision and ModifiedTime == DosDate + TimeSpan.FromTicks(0x800000). Add a Theory for extra-byte counts 1/2/3 mapping to HighPrecision1/HighPrecision2/NtfsPrecision.

- _high · untested-edge-case · confirmed_ — `ReScene.Lib/ReScene/RAR/RARHeaderReader.cs` :: ReadExtendedTimes lines 597-600

However there are concrete, high-impact gaps: (1) PatchEndOfArchiveCRC and the entire Archive-Data-CRC fixup path is NEVER exercised — every synthetic End-of-Archive block in the tests uses flags 0x4000, never the DataCRC flag 0x0002, so the most reconstruction-critical CRC fixup is untested; (2) AnalyzeFile silently omits ServiceBlockFileTime and FileModifiedTimes from its wouldModify computation, so it diverges from PatchStream and that divergence is untested; (3) FileModifiedTimes name matching is fed a filename read from a fixed offset 32 that is wrong for LARGE-flag headers, a latent correctness bug with no test. Truncated/malformed input, multi-volume (NextVolume) End-of-Archive, Unicode/long names, RAR5 rejection, cancellation, and idempotency (patch-twice) are also untested.

**Test-quality notes:** The mtime/EXT_TIME tests are the strongest in the file: they assert exact remainder bytes AND round-trip the patched archive through the real RARHeaderReader, which is the right way to validate bit-exact reconstruction. The LARGE-flag tests likewise re-parse and check CRCValid + HasLargeSize. Weaknesses: PatchFile_NoChangesNeeded_ReturnsEmptyResults (lines 100-138) has NO assertions at all — its body computes originalHostOS, never uses it, and the trailing comment admits nothing is verified; it is a no-op test. PatchFile_ServiceBlockFileTime_PatchesCorrectly (lines 157-173) only asserts NotEmpty and never checks the service block's FileTime field was actually zeroed. Several PatchFile/AnalyzeFile tests assert only Assert.NotEmpty without verifying the file bytes or re-parse validity. PatchStream is reached mostly indirectly through PatchFile; only one direct PatchStream test (combined HostOS+Attributes) exists, and none asserts the End-of-Archive CRC fixup. No test uses a real multi-file/multi-volume scene RAR with a DataCRC End-of-Archive block, so the realistic reconstruction path is validated only on synthetic single-file archives.

**Coverage assessment:** Coverage for this unit is genuinely strong overall. ### `rar-decompression` [lib] — 9 findings (1C / 3H / 4M / 1L)

**Coverage assessment:** Coverage is substantially better than the "~109 lines of tests" framing implies, because DecompressionTests.cs is not the only test source for this unit. ### ❌ BitInput.GetBits/GetBits32 use '>=' bounds check, dropping the last 1-2 readable bytes of the buffer  `(rar-decompression, was high)`
- The code is actually correct; this is not a demonstrated bug. GetBits (line 71) reads indices InAddr, InAddr+1, InAddr+2 and bails with `if (InAddr + 2 >= InBuf.Length) return 0`. That guard fires exactly when the highest index (InAddr+2) would be out of bounds, i.e. >= Length. At InAddr == Length-3, InAddr+2 == Length-1 < Length, the check is FALSE, and the full 3-byte read of the last three bytes proceeds correctly. At InAddr == Length-2 the read would need index Length (OOB), so returning 0 is the correct safe behavior. There is no 'one position too early' bail and no readable byte is dropped. GetBits32 (line 95) is analogously correct: it bails when InAddr+4 >= Length (index InAddr+4 OOB), and the extra guarded read on line 108 (`if (InAddr + 4 < InBuf.Length)`) correctly only pulls the 5th byte when it is in-bounds. The +2 vs +4 thresholds are not 'inconsistent' — they reflect that GetBits touches 3 bytes and GetBits32 touches up to 5. The finding's own description is self-contradictory (it admits the off-by-one 'is masked for short inputs') and never exhibits an input where the result is wrong. Since InBuf is always exactly MaxSize with no guaranteed trailing padding, the `>=` guard is the necessary safety bound, not an error. This is at most a coverage gap (boundary tests), already covered conceptually by finding 7's HuffmanDecoder/BitInput primitive-test recommendation; as a stated correctness bug it is unfounded.
